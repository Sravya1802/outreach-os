0. TL;DR — DOES MY MAC STAY ON?
================================================================================

NO.

The Mac is only involved during deployment:
  1. You run `gcloud run deploy` from your terminal
  2. Google pulls your source, builds a container in the cloud, deploys it
  3. Mac can be shut down — backend keeps running on Google's servers
  4. Only re-open Mac when you want to deploy new code

Once live, auto-apply runs entirely on Google Cloud Run:
  - User clicks "Apply" on Vercel frontend
  - Request hits Cloud Run URL → container spins up (if cold)
  - Playwright launches headless Chromium inside the container
  - Auto-apply completes, writes result to Supabase, returns response
  - Mac is not touched at any point

5. BACKEND CODE CHANGES NEEDED
================================================================================

5.1 SWAP SQLITE → POSTGRES (Supabase)

  Your backend currently uses better-sqlite3 writing to data/outreach.db.
  Cloud Run containers are EPHEMERAL — filesystem resets on every cold start,
  so SQLite cannot persist. Move to Supabase Postgres.

  Steps:
  (a) Sign up at https://supabase.com (free)
  (b) Create project "outreach-os"
  (c) Supabase Dashboard → Database → Connection string → copy "Transaction"
      connection (pooler, port 6543) — this is for serverless environments like
      Cloud Run
  (d) Add `pg` or `postgres` npm package to backend:
        npm install pg --prefix backend
  (e) Refactor backend/db.js:
        - Replace Database(path) → new Pool({ connectionString: DATABASE_URL })
        - Replace db.prepare().run/get/all → pool.query(sql, params)
        - Supabase Postgres dialect needs $1, $2 placeholders (not ?)
  (f) Run backend/db.js schema migrations via Supabase SQL Editor:
        - Copy CREATE TABLE statements from current db.js
        - Replace SQLite-specific syntax:
            INTEGER PRIMARY KEY AUTOINCREMENT → SERIAL PRIMARY KEY
            datetime('now')                  → NOW()
            TEXT DEFAULT ...                 → TEXT DEFAULT ...  (same)
        - Paste into Supabase SQL editor → Run

  Note: This migration is independent of Cloud Run. If you eventually land
  Oracle capacity, the Postgres migration still applies because Oracle ARM VM
  would also connect to Supabase Postgres, not SQLite.

5.2 ADD A /healthz ENDPOINT (for Cloud Run health checks)

  In backend/server.js, add:
    app.get('/healthz', (req, res) => res.status(200).send('ok'));

5.3 LISTEN ON $PORT (Cloud Run injects a PORT env var)

  In backend/server.js, change the final listen():
    const PORT = process.env.PORT || 3000;
    app.listen(PORT, '0.0.0.0', () => console.log(`Listening on ${PORT}`));

  The '0.0.0.0' bind is critical — Cloud Run rejects containers that bind to
  localhost only.

5.4 MOVE SECRETS OUT OF .env FILES

  Cloud Run reads env vars from deployment config, not from files.
  Keep using backend/.env for local dev, but deployment passes secrets via
  `--set-env-vars` at deploy time (covered in §7).

5.5 ENSURE PLAYWRIGHT USES THE CONTAINERIZED CHROMIUM

  Playwright must be installed with browsers INSIDE the container build.
  The Dockerfile (§6) handles this — no code change needed if you use
  playwright.chromium.launch() without a hardcoded executablePath.

6. DOCKERFILE — BUILDS YOUR BACKEND INTO A CONTAINER
================================================================================

Create `/backend/Dockerfile` with these contents:

  # Multi-stage build: Playwright base image has Chromium + all deps baked in
  FROM mcr.microsoft.com/playwright:v1.59.1-jammy AS runtime

  WORKDIR /app

  # Copy package manifests first for better Docker layer caching
  COPY package.local.json ./package.json
  COPY package-lock.local.json ./package-lock.json

  # Install only production deps
  RUN npm ci --only=production

  # Copy the rest of the backend source
  COPY . .

  # Cloud Run injects $PORT
  ENV PORT=8080
  EXPOSE 8080

  # Playwright browsers are already installed in the base image, but confirm:
  RUN npx playwright install chromium --with-deps

  # Non-root user for security (optional but recommended)
  USER pwuser

  CMD ["node", "server.js"]


Create `/backend/.dockerignore` with:

  node_modules
  npm-debug.log
  .env
  .env.*
  data/
  *.db
  *.sqlite
  output/
  logs/
  .git
  .DS_Store


Why this base image:
  - mcr.microsoft.com/playwright:v1.59.1-jammy = Ubuntu 22.04 + Node 20
    + Playwright 1.59 + full Chromium with all fonts/libs pre-installed
  - Saves ~4 minutes of build time per deploy (no apt-get install)
  - Matches your backend's playwright version exactly
  - Works on Cloud Run's Linux AMD64 runtime out of the box

7. DEPLOYMENT — STEPS TO GO LIVE
================================================================================

First-time deploy (from your Mac, inside the backend/ directory):

  cd "/Users/lakshmisravyarachakonda/VS CODE/email tracker/backend"

  gcloud run deploy outreach-backend \
    --source . \
    --region us-central1 \
    --platform managed \
    --allow-unauthenticated \
    --memory 2Gi \
    --cpu 1 \
    --timeout 3600 \
    --max-instances 3 \
    --min-instances 0 \
    --concurrency 10 \
    --set-env-vars "NODE_ENV=production,DATABASE_URL=<supabase-connection-string>,ANTHROPIC_API_KEY=...,OPENAI_API_KEY=...,GEMINI_API_KEY=...,APIFY_TOKEN=..." \
    --project outreach-os-<suffix>


What each flag does:
  --source .              Builds from current directory using Cloud Build
  --allow-unauthenticated Public endpoint (your frontend calls it directly)
  --memory 2Gi            2 GB RAM per instance (Playwright needs this)
  --cpu 1                 1 vCPU — you can bump to 2 if needed
  --timeout 3600          60-min per-request timeout (max allowed)
  --max-instances 3       Hard cap: no more than 3 concurrent containers
  --min-instances 0       Scales to zero when idle (= $0 at idle)
  --concurrency 10        Each container handles up to 10 concurrent requests
  --set-env-vars ...      All secrets passed at deploy time (safer than .env)


Output shows your Cloud Run URL, e.g.:
  https://outreach-backend-abc123-uc.a.run.app


Test it:
  curl https://outreach-backend-abc123-uc.a.run.app/healthz
  → should return "ok"


Subsequent deploys (after code changes):
  gcloud run deploy outreach-backend --source . --region us-central1
  (env vars persist from previous deploy unless you re-set them)

9. SCHEDULED JOBS — CLOUD SCHEDULER (replaces node-cron)
================================================================================

Your backend uses node-cron. On Cloud Run, the container scales to zero when
idle, so in-process cron does NOT work. Use Cloud Scheduler to trigger
endpoints on a schedule.

Example: run auto-apply daily at 9 AM:

  gcloud scheduler jobs create http daily-autoapply \
    --location us-central1 \
    --schedule "0 9 * * *" \
    --time-zone "America/Chicago" \
    --uri "https://outreach-backend-abc123-uc.a.run.app/career/auto-apply/run" \
    --http-method POST \
    --body '{"mode":"auto","limit":10}' \
    --headers "Content-Type=application/json" \
    --project outreach-os-<suffix>


Repeat for any other cron-triggered routes (scraping, cleanup, etc.).
Free tier = 3 jobs. Combine cron triggers into one endpoint if you need more.

12. COMMON PITFALLS AND FIXES
================================================================================

P1. "Container failed to start" on first deploy
   Cause: binding to 127.0.0.1 instead of 0.0.0.0
   Fix: ensure app.listen(PORT, '0.0.0.0', ...)

P2. Playwright error: "browserType.launch: Executable doesn't exist"
   Cause: Playwright browsers not installed in container
   Fix: ensure Dockerfile has RUN npx playwright install chromium --with-deps
        (or use the mcr.microsoft.com/playwright base image as in §6)

P3. "Memory limit exceeded"
   Cause: Chromium + your app exceeded 1 GB
   Fix: --memory 2Gi on deploy (still free-tier)

P4. Slow first request after idle (~3–5s cold start)
   Cause: Cloud Run scales to zero, new container takes time to boot
   Fix (if needed): --min-instances 1
     WARNING: This keeps one instance always running = uses continuous GB-sec
     and will push you into paid territory. Only use if UX requires it.

P5. "Could not connect to Supabase"
   Cause: Cloud Run cannot reach Supabase direct connection (port 5432)
   Fix: use Supabase's POOLER connection on port 6543 (transaction mode)

P6. Deploy hangs at "Building using Dockerfile..."
   Cause: .dockerignore missing, Cloud Build uploads 500 MB node_modules
   Fix: verify .dockerignore excludes node_modules and data

P7. 404 on API calls from frontend
   Cause: VITE_API_BASE_URL not set or frontend not redeployed
   Fix: redeploy Vercel after setting the env var, confirm in browser devtools

P8. Scheduled job never fires
   Cause: Cloud Scheduler wrong region, wrong URL, or auth missing
   Fix: run `gcloud scheduler jobs run <name>` manually to see the error

13. HYBRID STRATEGY — IF ORACLE LANDS LATER
================================================================================

If your Oracle retry script eventually lands a VM AFTER you've already
deployed to Cloud Run:

Option A: Keep Cloud Run, ignore Oracle
  - You're already running $0 on Cloud Run with less maintenance
  - Terminate the Oracle VM to free Always-Free quota for future use

Option B: Migrate to Oracle for more RAM / persistent compute
  - Set up Oracle VM per original PLAN.txt Phase 2
  - Point Vercel VITE_API_BASE_URL at Oracle IP instead of Cloud Run
  - Decommission Cloud Run: gcloud run services delete outreach-backend
  - Advantage: no cold starts, persistent filesystem
  - Disadvantage: you manage OS patches, SSL renewal, nginx config

Decision rule:
  - Low traffic (where you are now): Cloud Run wins on simplicity
  - High traffic or need persistent compute: Oracle VM wins on capacity


================================================================================
14. EXIT CRITERIA — WHEN TO ABANDON PLAN B
================================================================================

Stop and pivot if ANY of these happen:

  - Monthly bill exceeds $5 unexpectedly
    Likely cause: min-instances set > 0, or a runaway loop in your code
    Action: set --min-instances 0, investigate logs for infinite loops

  - Cold starts unacceptable for UX (reports from users)
    Action: increase --min-instances to 1 (accept small monthly cost ~$3)
            OR pivot to Oracle VM if capacity has landed by then

  - Feature you need is not supported
    Unlikely — Cloud Run is a full Linux container, supports almost everything

  - Supabase free tier exceeded (500 MB DB)
    Action: upgrade Supabase to Pro ($25/mo) OR migrate DB to a cheaper
            provider (Neon free, PlanetScale, self-hosted on Oracle VM
            if available)
